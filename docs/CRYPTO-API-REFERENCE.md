# 📚 مرجع API — پرداخت یکپارچه و ارز دیجیتال

این فایل مکمل [`API-REFERENCE.md`](./API-REFERENCE.md) هست (که فقط کارت‌به‌کارت رو پوشش می‌ده). اینجا endpointهای مربوط به **ارز دیجیتال** و **روتر یکپارچه** (کارت + کریپتو، انتخاب با مشتری) توضیح داده می‌شه.

> 📌 اگه فقط از ووکامرس استفاده می‌کنید، لازم نیست این فایل رو بخونید — کافیه تو تنظیمات افزونه، «آدرس سرور یکپارچه» رو هم پر کنید. این مستندات برای اتصال مستقیم (بدون افزونه) یا ساخت اتصال سفارشیه.

---

## 🚀 endpoint پیشنهادی: ساخت سفارش یکپارچه

**Endpoint:**
```
POST https://cubevps.ir/pay/create-order.php
```

بسته به تنظیمات فروشنده («⚙️ تنظیمات بیشتر → 💳 روش‌های پرداخت» تو ربات)، این endpoint خودش تصمیم می‌گیره:
- فقط کارت فعاله → مستقیم فاکتور کارتی می‌سازه و لینکش رو می‌ده
- فقط کریپتو فعاله → مستقیم فاکتور کریپتویی می‌سازه
- هر دو فعاله → یه لینک به صفحه‌ی «کارت یا کریپتو؟» می‌ده؛ مشتری انتخاب می‌کنه و از همون‌جا به فاکتور واقعی هدایت می‌شه

### 📋 پارامترها

| نام | نوع | اجباری | توضیحات |
|---|---|---|---|
| `order_id` | string | ✅ | شناسه‌ی یکتای سفارش شما |
| `price_amount` | number | ✅ | مبلغ به **تومان** (نه ریال — برخلاف endpoint کارتیِ قدیمی) |
| `callback_url` | string | ❌ | آدرسی که بعد از نتیجه‌ی نهایی (چه کارتی چه کریپتویی) بهش خبر داده می‌شه |

### ✅ نمونه پاسخ

```json
{
  "success": true,
  "method": "card",
  "pay_page_url": "https://cubevps.ir/smspay/pay.php?authority=...",
  "authority": "bdc9e0497c..."
}
```

مقدار `method` می‌تونه `card`, `crypto`, یا `choice` باشه. برای `choice`، فیلد `authority` وجود نداره (چون هنوز روش مشخص نشده) — نتیجه‌ی نهایی رو فقط از طریق `callback_url` می‌فهمید.

## 🪙 endpointهای مستقیمِ ارز دیجیتال (اگه نمی‌خواید از روتر یکپارچه استفاده کنید)

### ساخت فاکتور کریپتویی

```
POST https://cubevps.ir/crypto/api/create-crypto-payment.php
```

| نام | نوع | اجباری | توضیحات |
|---|---|---|---|
| `order_id` | string | ✅ | شناسه‌ی یکتای سفارش شما |
| `price_amount` | number | ✅ | مبلغ به **تومان** |
| `callback_url` | string | ❌ | آدرس اطلاع‌رسانیِ نتیجه‌ی نهایی |

مشتری خودش تو صفحه‌ی برگشتی (`pay_page_url`) ارز (USDT-BEP20 / TRX / TON) رو انتخاب می‌کنه — این endpoint فوراً آدرس واریز نمی‌سازه، فقط یه فاکتور «در انتظار انتخاب ارز» می‌سازه.

### چک وضعیت (Polling، اختیاری)

```
GET https://cubevps.ir/crypto/api/check-crypto-payment-status.php?payment_id=XXXX
```
`payment_id` رو از callback یا از polling سمت مشتری (تو همون صفحه‌ی پرداخت) به دست میارید.

## 🔄 اطلاعات ارسالی به `callback_url`

### برای پرداخت کارتی
دقیقاً مثل [`API-REFERENCE.md`](./API-REFERENCE.md#-اطلاعات-ارسالی-به-callback_url) — شامل `authority`. برای تایید نهایی، `verify-payment.php` رو صدا بزنید.

### برای پرداخت کریپتویی
```json
{
  "success": true,
  "status": "paid",
  "order_id": "ORD123",
  "payment_id": "4546796864",
  "pay_currency": "usdtbsc",
  "amount": "0.26",
  "sig": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
}
```
(هم به‌صورت POST با بدنه‌ی JSON، هم querystring GET روی همون آدرس)

مقدار `status` می‌تونه `paid`, `failed`, یا `expired` باشه.

### ⚠️ حتماً امضا (`sig`) رو چک کنید

چون این کال‌بک به سایت شما (که یه آدرس عمومیه) می‌رسه، هرکسی می‌تونه تئوری یه درخواست جعلی به همون آدرس بفرسته. برای اطمینان از اصالتش، `sig` رو با کلید API token خودتون دوباره بسازید و مقایسه کنید:

```php
$expectedSig = hash_hmac('sha256', $orderId . '|' . $status . '|' . $amount, $apiToken);
if (!hash_equals($expectedSig, $sig)) {
    // جعلیه — نادیده بگیرید
    exit;
}
```

اگه امضا مطابقت نداشت، اون درخواست رو کاملاً نادیده بگیرید و سفارش رو «پرداخت‌شده» نکنید.

## ⚠️ نکات مهم

- برخلاف endpointهای کارتی (که مبلغ رو به **ریال** می‌گیرن)، همه‌ی endpointهای کریپتویی/یکپارچه مبلغ رو به **تومان** می‌گیرن.
- هر پرداخت (کارتی یا کریپتویی) **۶۰ دقیقه** اعتبار داره.
- هر ارز کریپتو یه حداقل مبلغ مجاز داره (چند ده هزار تومان بسته به نرخ روز) — اگه مبلغ کمتر باشه، پیام خطای روشن برمی‌گرده.
- کارمزد پلتفرم از پرداخت‌های کریپتویی به‌صورت درصدی محاسبه می‌شه (نه مبلغ ثابت تومانی مثل کارتی).
